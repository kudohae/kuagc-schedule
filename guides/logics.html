A-3. 요청(request) 처리 로직

타입승인 시거절 시
absent slot_exceptions에 {exception_type:'absent', week_offset, day, hour} 삽입. 이미 absent exception이 있으면 중복 삽입 건너뜀. requests.status = 'rejected'만 업데이트. exception 미생성.
extra slot_exceptions에 {exception_type:'extra'} 삽입
terminate 해당 팀의 base_slots에서 day/hour 일치 레코드 영구 DELETE + 기존 exception도 삭제. (예외가 아닌 base_slots 자체를 수정하는 유일한 경우)

A-4. 시간 배정 회차 로직

open → close_at 도달 또는 수동 마감 → closed → 배정 실행 + 초안 승인 → finished
배정 알고리즘 (schedule.js → assignSlots)
  1. round_id에 해당하는 신청 중 team_id별 최신(submitted_at) 1건만 추출 (마지막 제출이 이전 제출 무효화)
  2. submitted_at 오름차순 정렬 (선착순 기준)
  3. 1지망 순회 — occupied Set에 없으면 배정 + Set에 추가
  4. 1지망 미배정 → 2지망 순회 (동일 방식)
  5. 2지망 미배정 → 3지망 순회
  6. 결과를 time_applications.assigned_day/hour/pref에 일괄 UPDATE
초안 승인 (schedule.js → approveDraft)
  1. base_slots에서 해당 season의 모든 레코드 DELETE
  2. assigned_day/hour가 not null인 applications에서 {team_id, day, hour, season} 추출
  3. base_slots에 INSERT
  4. application_rounds.draft_approved = true로 UPDATE
예약 오픈/마감: open_at / close_at이 설정된 경우, 클라이언트에서 setInterval(1s)로 체크. new Date(open_at) ≤ Date.now() 이면 자동으로 status UPDATE.

B-3. 마감 후 배정 로직 (admin.html → adminSchoolClose)

  2. status = 'pending'인 applications를 created_at ASC로 정렬 (선착순)
  3. 각 pending app에 대해 pref1 → pref2 순으로 가용 여부 체크 후 배정
  4. 내부 counts 맵을 루프 내에서 실시간 갱신 (이미 처리된 배정도 반영)
  5. 각 결과를 school_applications UPDATE
counts 맵 초기값: 이미 assigned 상태인 레코드 수로 초기화. 배정 확정 시마다 counts[schoolId]++.

C-7. DnD 팀 구성 초기 상태 계산 (computeEnsDndInitial)

slot 구조: { id: "${appId}-${session}", appId, session, applicantName, studentId, createdAt, songTitle, sessionRound, isApplicant }

isApplicant 판별: session_app.student_id === song_app.student_id (곡 신청자 본인 여부)
우선순위경로처리
경로 1 localStorage 임시저장 있음 locMap { slotId → songId | 'pool' }으로 위치 복원. 단, isApplicant = true인 slot은 locMap 무시하고 항상 자신의 곡에 배치.
경로 2 이미 confirmed/rejected 앱 있음 confirmed → 해당 곡에 배치. pending + isApplicant = true → 해당 곡에 배치. pending + isApplicant = false → unassigned pool.
경로 3 최초 DnD 열기 created_at ASC 정렬. isApplicant = true → 무조건 slots + claimed 등록. 나머지 → claimed에 해당 session 없으면 slots, 있으면 pool.
충돌 감지 (getConflictedSlotIds): 동일 곡에서 동일 session을 두 명 이상이 담당하면 Set<slotId> 반환 → 주황색으로 표시.

C-8. DnD 확정 로직 (confirmEnsembleTeams)

  1. eDndSt.songs에 배치된 slots를 appId별로 그룹핑: appAssign = { appId: { songId, sessions: Set } }
  2. 같은 appId가 여러 곡에 걸쳐 있으면 가장 많은 sessions를 가진 곡을 best로 선택
  3. 각 appId에 대해 DB UPDATE: session_applications SET status='confirmed', song_id=best.songId, sessions=[...best.sessions]
  4. unassigned pool에 남은 slot들의 appId → status='rejected'
  5. phase는 변경하지 않음 (DnD 확정 ≠ 단계 전환)
슬롯 ID 파싱: slot.id = "${appId}-${session}" 이므로 appId = parseInt(slotId.split('-')[0])
